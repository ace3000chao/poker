## 1. 插件是什么:核心概念

一个游戏插件 = `backend/games/` 下的一个**独立目录**。平台启动时自动扫描该目录、
加载插件、把它的接口挂到 `/api/games/<目录名>/` 路径下。

### 1.1 平台与插件的分工(最重要的一条)

| 职责 | 谁来做 |
|------|--------|
| 用户登录、签发 JWT | **平台** |
| 校验登录态(`Authorization` 头) | **平台**(装饰器) |
| 校验游戏是否上架、是否超出每日次数 | **平台**(公共服务层) |
| 计算积分、写积分、更新排行榜 | **平台**(公共服务层) |
| 游戏玩法逻辑、判定胜负、产出原始成绩 | **插件** |

**插件只负责玩法,不碰积分、不碰用户表。** 插件把"这一局得了多少分、是否获胜"
上报给平台,由平台统一结算积分。这是平台的硬性设计(技术决策 Q6),不要绕过它。

### 1.2 自动注册:无需改动平台代码

你**不需要**修改 `backend/app.py` 或任何平台文件。把插件目录放进 `backend/games/`,
重启服务即可。平台的 `games/registry.py` 会:

1. 扫描 `backend/games/` 每个子目录,读取其 `config.json`;
2. 导入子目录的 `__init__.py`,取出名为 `bp` 的 Blueprint,挂到 `/api/games/<目录名>/`;
3. 把插件信息写入数据库 `games` 表(首次出现时按 `config.json` 的 `enabled` 决定是否上架)。

> 某个插件加载失败会被隔离,只记录日志,不影响平台和其它插件。

### 1.3 目录名就是 game_id

平台用**目录名**作为这个游戏的唯一标识(`game_id`):

- 路由前缀:`/api/games/<目录名>/`
- 数据库 `games.game_id` 字段
- 公共服务层按目录名查询游戏

因此:**目录名 = `config.json` 里的 `id` 字段 = 你代码里的 `_GAME_KEY` 常量**,三者必须完全一致。

目录名规则:只用小写英文字母、数字、下划线;全局唯一,不能和已有插件重名。
例:`card_guess`、`poker_quiz`、`memory_match`。

## 2. 快速开始:5 步做一个最小插件

假设我们做一个叫 `card_guess` 的插件。

**第 1 步**　建目录与文件:

```
backend/games/card_guess/
├── __init__.py      # 暴露 bp
├── config.json      # 游戏配置
└── routes.py        # 三个标准路由
```

**第 2 步**　`config.json`(字段详解见第 3.2 节):

```json
{
  "id": "card_guess",
  "name": "猜花色",
  "version": "1.0.0",
  "description": "翻开一张校友牌,猜它的花色",
  "icon": "static/icon.png",
  "enabled": true,
  "settings": {
    "max_daily_games": 10,
    "points_per_win": 10,
    "points_per_loss": 2,
    "time_limit_seconds": 60
  },
  "score_schema": {
    "score": "integer",
    "duration": "integer",
    "timestamp": "string",
    "is_win": "boolean"
  }
}
```

**第 3 步**　`__init__.py` —— 必须暴露名为 `bp` 的 Blueprint:

```python
"""猜花色 游戏插件。"""
from .routes import bp

__all__ = ["bp"]
```

**第 4 步**　`routes.py` —— 实现 `/info`、`/play`、`/score` 三个路由(完整版见第 7 节)。

**第 5 步**　重启后端 `flask run`,看启动日志出现 `已注册游戏插件:[..., 'card_guess']`
即接入成功。访问 `GET /api/games/card_guess/info` 验证。

## 3. 文件契约

### 3.1 目录结构

```
backend/games/<game_id>/
├── __init__.py     必填  暴露名为 bp 的 Flask Blueprint
├── config.json     必填  游戏配置
├── routes.py       必填  三个标准路由的实现
├── static/         可选  静态资源(图标等),需在 Blueprint 中开启 static_folder
│   └── icon.png
└── models.py       可选  插件私有数据表(进阶,见第 6.4 节)
```

`__init__.py` 的唯一硬性要求:模块顶层有一个名为 `bp` 的 `flask.Blueprint` 对象。
平台用 `getattr(module, "bp")` 取它;取不到则跳过该插件并打印警告。

### 3.2 config.json 字段

| 字段 | 类型 | 必填 | 说明 |
|------|------|:--:|------|
| `id` | string | ✅ | 游戏标识,**必须等于目录名** |
| `name` | string | ✅ | 显示名称 |
| `version` | string | ✅ | 版本号,`x.y.z` |
| `description` | string | ✅ | 简介,50 字以内 |
| `icon` | string | ✅ | 图标路径,相对插件目录 |
| `enabled` | boolean | ✅ | **仅首次入库时**决定是否上架(见 3.3) |
| `settings` | object | ✅ | 玩法参数,见下表 |
| `score_schema` | object | ✅ | 成绩上报字段说明(供前端/文档参考,平台不强校验) |

`settings` 字段(平台计分时会读取):

| 字段 | 类型 | 缺省 | 说明 |
|------|------|:--:|------|
| `max_daily_games` | integer | 10 | 每个用户每天最多玩几次 |
| `points_per_win` | integer | 10 | 获胜时平台发放的积分 |
| `points_per_loss` | integer | 2 | 失败时平台发放的积分 |
| `time_limit_seconds` | integer | 300 | 单局时间限制(秒),由插件自行执行 |

你可以在 `settings` 里加自定义字段,平台会原样保留,你的插件代码可自行读取。

### 3.3 `enabled` 的真实语义(易踩坑)

`config.json` 的 `enabled` **只在插件第一次被平台发现、写入 `games` 表时**生效。
之后上架/下架状态以数据库 `games.is_enabled` 为准,由**管理员后台**控制,
不会被 `config.json` 再次覆盖。

含义:
- 你交付插件时设 `"enabled": true`,首次部署即可玩;设 `false` 则需管理员后台手动上架。
- 插件已上线后,改 `config.json` 的 `enabled` **不会**改变线上状态。下架请走管理员后台。
- 一个插件即使 Blueprint 注册成功,只要 `games.is_enabled` 为假,`/play` 和 `/score`
  都会返回 `40402(游戏已下架)`。

## 4. 三个标准路由

所有路由都挂在 `/api/games/<game_id>/` 下。平台统一响应信封见第 5.1 节。

### 4.1 `GET /info` —— 游戏信息(公开,无需登录)

返回游戏基本信息,供前端展示。**不要**加登录校验。

响应 `data`:

```json
{
  "id": "card_guess",
  "name": "猜花色",
  "description": "翻开一张校友牌,猜它的花色",
  "icon": "/api/games/card_guess/static/icon.png",
  "settings": { "max_daily_games": 10, "points_per_win": 10,
                "points_per_loss": 2, "time_limit_seconds": 60 }
}
```

### 4.2 `POST /play` —— 开始一局(需登录)

用户真正开始一局时调用。**必须**在函数里调 `scoring.start_session()`,
它会校验"游戏是否下架 / 是否超每日次数",并把当日次数 +1。

> 一次 `/play` 就消耗一次每日次数。请在游戏真正开始时调用,不要在进入页面时调用。

请求:`Authorization: Bearer <access_token>`,无 body。

响应 `data`(字段由插件决定,以下为约定示例):

```json
{
  "game_session_id": "sess_3f9a1c2b7e04",
  "started_at": "2026-05-21T08:00:00+00:00",
  "time_limit": 60
}
```

> `game_session_id` 由插件自行生成。当前平台**不持久化、不校验**它。
> 若你的玩法需要防作弊的会话校验,需由插件自己实现(例如把 session 存内存/表)。

可能错误码:`40101` 未登录、`40102` Token 过期、`40401` 游戏不存在、
`40402` 游戏已下架、`42902` 今日次数已用完。

### 4.3 `POST /score` —— 上报成绩(需登录)

一局结束时上报原始成绩。**必须**调 `scoring.submit_score()`,由平台结算积分。

请求:`Authorization: Bearer <access_token>`,`Content-Type: application/json`,body:

| 字段 | 类型 | 必填 | 说明 |
|------|------|:--:|------|
| `score` | integer | ✅ | 本局原始得分 |
| `duration` | integer | ✅ | 本局时长(秒) |
| `timestamp` | string | ✅ | 结束时间,ISO 8601 |
| `is_win` | boolean | 可选 | 是否获胜。**不传则平台按 `score > 0` 判定** |
| `cards_used` | array | 可选 | 本局用到的牌 key 列表,如 `["hearts_A"]` |

> 必填字段由**插件路由自己校验**(见 demo_game:缺字段返回 `40001`)。平台不强校验 body。

响应 `data`:

```json
{
  "earned_points": 10,
  "total_points": 150,
  "rank": 42,
  "is_win": true
}
```

平台据 `is_win` 决定发 `points_per_win` 还是 `points_per_loss`,累加到用户总积分,
写入 `game_scores` 表,并返回该用户在全局积分榜的名次。

可能错误码:`40001` 参数错误、`40101` 未登录、`40102` Token 过期、
`40401` 游戏不存在、`40402` 游戏已下架、`42903` 上报间隔太短。

> **防刷**:同一用户对同一游戏两次 `/score` 间隔须 ≥ 10 秒,否则返回 `42903`。

### 4.4 静态资源(图标等)

平台**不会**自动托管插件的 `static/` 目录。若 `/info` 要返回可访问的图标 URL,
请在创建 Blueprint 时开启静态目录:

```python
bp = Blueprint("card_guess", __name__,
               static_folder="static", static_url_path="/static")
```

这样 `static/icon.png` 即可经 `/api/games/card_guess/static/icon.png` 访问。

## 5. API 调用规范

### 5.1 统一响应信封

平台所有接口(含插件接口)都返回同一结构,HTTP 状态码通常恒为 `200`,
**业务结果看 `code` 字段**:

```json
{ "code": 0, "message": "success", "data": { } }
```

- `code == 0`:成功,数据在 `data`。
- `code != 0`:失败,`message` 是中文说明,`data` 为 `null`。

插件路由里请用平台 `errors.py` 的 `ok(data)` / `fail(code, message)` 构造响应,
不要自己拼 JSON。

### 5.2 鉴权:如何带 Token

`/play`、`/score` 需要登录。调用方在请求头带:

```
Authorization: Bearer <access_token>
```

平台用 `@require_auth` 装饰器校验,通过后把当前用户对象放进 Flask 的
`g.current_user`,你在路由里直接用 `g.current_user`。

### 5.3 本地调试时怎么拿到 Token

平台采用"手机号 + 短信验证码"登录。本地开发环境**短信是桩实现**——
验证码不会真的发出,而是打印在 Flask 运行日志里。流程:

```bash
# 1. 请求验证码(看后端终端日志里的 [SMS-STUB] code=XXXXXX)
curl -X POST http://localhost:5000/api/auth/send-code \
     -H 'Content-Type: application/json' \
     -d '{"phone":"13800138000"}'

# 2. 用日志里的验证码登录,响应 data.access_token 即为 Token
curl -X POST http://localhost:5000/api/auth/login \
     -H 'Content-Type: application/json' \
     -d '{"phone":"13800138000","code":"日志里的6位码"}'
```

新手机号首次登录会自动注册。拿到 `access_token` 后即可调用插件接口。

## 6. 平台公共服务层 API(插件必用)

插件不直接读写 `users` / `game_scores` 表,而是调用 `common/scoring.py` 提供的函数。

### 6.1 `scoring.start_session(user, game_key)`

在 `/play` 里调用。`user` 传 `g.current_user`,`game_key` 传你的目录名常量。

行为:校验游戏存在且已上架、校验未超每日次数、当日次数 +1。
返回该游戏的 `config.json` 字典(可读 `time_limit_seconds` 等)。
失败时抛 `ScoringError`。

### 6.2 `scoring.submit_score(user, game_key, payload)`

在 `/score` 里调用。`payload` 传 `/score` 收到的 body 字典。

行为:校验上报间隔(≥10s)、按 `settings` 结算积分、写 `game_scores`、
累加用户积分。返回 `{earned_points, total_points, rank, is_win}`。
失败时抛 `ScoringError`。

### 6.3 错误处理模式

公共服务层用 `ScoringError` 异常携带错误码。插件路由统一这样转译:

```python
from common import scoring
from common.scoring import ScoringError
from errors import ok, fail

try:
    result = scoring.submit_score(g.current_user, _GAME_KEY, body)
except ScoringError as e:
    return fail(e.code, e.message)
return ok(result)
```

### 6.4 插件私有数据表(进阶,默认不建议)

MVP 阶段建议插件保持**服务端无状态**:把每局需要留存的数据放进 `/score` 的
`cards_used` / 自定义字段,由平台写入 `game_scores`。

如果玩法确实需要自己的表,可加 `models.py` 定义模型,但要注意:平台的
Flask-Migrate 迁移**只覆盖核心 8 张表**,不会自动迁移你的插件表。你需要自行
处理建表与迁移,并避免与核心表名冲突。接入前请先与平台维护者沟通。

## 7. 完整示例:card_guess 插件

可直接复制改造。三个文件构成一个能跑通的插件。

`backend/games/card_guess/__init__.py`:

```python
"""猜花色 游戏插件。"""
from .routes import bp

__all__ = ["bp"]
```

`backend/games/card_guess/config.json`:见第 2 节第 2 步。

`backend/games/card_guess/routes.py`:

```python
"""猜花色 —— 游戏插件标准路由。

实现 /info /play /score 三个标准路由,并接入平台公共服务层。
插件自身不写积分、不碰用户表,只上报原始成绩。
"""
import json
import os
import uuid
from datetime import datetime, timezone

from flask import Blueprint, request, g

from errors import ok, fail, ERR_PARAM
from auth.decorators import require_auth
from common import scoring
from common.scoring import ScoringError

# 目录名 = game_id = config.json 的 id,三者必须一致
_GAME_KEY = "card_guess"
_CONFIG_PATH = os.path.join(os.path.dirname(__file__), "config.json")

bp = Blueprint(_GAME_KEY, __name__,
               static_folder="static", static_url_path="/static")


def _load_config():
    with open(_CONFIG_PATH, "r", encoding="utf-8") as f:
        return json.load(f)


@bp.get("/info")
def info():
    """公开:返回游戏信息,不加登录校验。"""
    cfg = _load_config()
    return ok({
        "id": cfg["id"],
        "name": cfg["name"],
        "description": cfg["description"],
        "icon": f"/api/games/{cfg['id']}/static/icon.png",
        "settings": cfg["settings"],
    })


@bp.post("/play")
@require_auth
def play():
    """开始一局:平台校验下架/每日次数并 +1。"""
    try:
        cfg = scoring.start_session(g.current_user, _GAME_KEY)
    except ScoringError as e:
        return fail(e.code, e.message)
    return ok({
        "game_session_id": f"sess_{uuid.uuid4().hex[:12]}",
        "started_at": datetime.now(timezone.utc).isoformat(),
        "time_limit": cfg["settings"]["time_limit_seconds"],
    })


@bp.post("/score")
@require_auth
def score():
    """上报成绩:平台统一结算积分。"""
    body = request.get_json(silent=True) or {}
    # 必填字段由插件自己校验
    for field in ("score", "duration", "timestamp"):
        if field not in body:
            return fail(ERR_PARAM, f"缺少必填字段:{field}")
    try:
        result = scoring.submit_score(g.current_user, _GAME_KEY, body)
    except ScoringError as e:
        return fail(e.code, e.message)
    return ok(result)
```

## 8. 本地开发与调试

### 8.1 跑起平台后端

```bash
cd backend
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
flask db upgrade            # 建表
FLASK_ENV=dev python3 seed.py   # 灌 52 张牌占位数据(玩法用到牌时需要)
FLASK_ENV=dev flask run         # 启动,默认 :5000
```

把你的插件目录放进 `backend/games/` 后**重启** `flask run`,
启动日志出现 `已注册游戏插件:[...]` 列表里有你的 `game_id` 即注册成功。

### 8.2 冒烟测试

```bash
# 健康检查
curl http://localhost:5000/api/health

# 游戏信息(公开)
curl http://localhost:5000/api/games/card_guess/info

# 开始一局(需 Token,获取方式见 5.3)
curl -X POST http://localhost:5000/api/games/card_guess/play \
     -H "Authorization: Bearer <access_token>"

# 上报成绩
curl -X POST http://localhost:5000/api/games/card_guess/score \
     -H "Authorization: Bearer <access_token>" \
     -H 'Content-Type: application/json' \
     -d '{"score":80,"duration":12,"timestamp":"2026-05-21T08:00:00Z","is_win":true}'
```

### 8.3 常见问题

| 现象 | 原因 |
|------|------|
| 启动日志没有你的 game_id | 目录缺 `config.json`,或 `__init__.py` 没暴露 `bp` |
| 调 `/play` 返回 `40402` | 游戏未上架:`games.is_enabled` 为假,需管理员后台上架 |
| 调 `/play` 返回 `40401` | `_GAME_KEY` / `config.json` 的 `id` 与目录名不一致 |
| 调 `/play`/`/score` 返回 `40101` | 没带 `Authorization: Bearer` 头,或 Token 无效 |
| 调 `/score` 返回 `42903` | 两次上报间隔不足 10 秒 |
| `/info` 的图标 404 | Blueprint 没开 `static_folder`(见 4.4) |

## 9. 上架流程

1. 按本文开发并自测通过。
2. 把插件目录交付给平台维护者,部署进生产 `backend/games/`。
3. 重启后端,平台自动发现插件并在 `games` 表登记。
4. 管理员在后台(`/admin` → 游戏管理)将该游戏设为上架。
5. 上架后用户即可进入游戏、计分、上榜。

## 10. 错误码表

平台使用**全局唯一分段错误码**(源:`backend/errors.py`)。插件接口可能涉及:

| code | 含义 | 触发场景 |
|------|------|----------|
| `0` | 成功 | —— |
| `40001` | 参数格式错误 | `/score` 缺必填字段等 |
| `40002` | 业务逻辑错误 | 自定义业务校验失败 |
| `40101` | 未登录或 Token 无效 | 缺 `Authorization` 头 / Token 错误 |
| `40102` | Token 已过期 | Access Token 超 24h |
| `40301` | 无访问权限 | —— |
| `40401` | 游戏不存在 | game_id 对不上 / 未登记 |
| `40402` | 游戏已下架 | `games.is_enabled` 为假 |
| `42902` | 今日游戏次数已用完 | 超 `max_daily_games` |
| `42903` | 成绩上报间隔太短 | 两次 `/score` 间隔 < 10s |
| `50001` | 服务器内部错误 | —— |

> 错误码全局唯一、分段管理(`400xx` 参数/业务、`401xx` 认证、`403xx` 权限、
> `404xx` 资源、`423xx` 锁定、`429xx` 限流、`500xx` 服务器)。
> 插件**不要**自定义错误码数字;需要新错误码时联系平台维护者在 `errors.py` 登记。

## 11. 给插件开发者的约束清单

- ✅ 目录名 = `config.json` 的 `id` = 代码里的 `_GAME_KEY`,三者一致。
- ✅ `__init__.py` 暴露名为 `bp` 的 Blueprint。
- ✅ 实现 `/info`(公开)、`/play`、`/score`(需登录)三个路由。
- ✅ `/play` 调 `scoring.start_session`,`/score` 调 `scoring.submit_score`。
- ✅ 用 `ok()` / `fail()` 构造响应,用 `ScoringError` 错误码转译。
- ❌ 不要直接读写 `users`、`game_scores`、`daily_game_counts` 表。
- ❌ 不要自己计算/发放积分——积分一律由平台结算。
- ❌ 不要修改 `backend/app.py` 或任何平台核心文件。
- ❌ 不要复用/自造全局错误码数字。
