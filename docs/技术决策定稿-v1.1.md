## Q1. 全局唯一错误码表(取代各文档分散且冲突的错误码)

各旧文档中 `42901`/`42902` 在多场景被复用,**作废**,统一为下表(已在
`backend/errors.py` 实现):

| 段 | 错误码 | 说明 |
|----|--------|------|
| 成功 | `0` | success |
| 400xx | `40001` | 参数格式错误 |
| | `40002` | 业务逻辑错误(如 session 不匹配) |
| | `40003` | 验证码错误 |
| | `40004` | 验证码已过期 |
| | `40005` | 验证码已使用 |
| | `40006` | 验证码用途不匹配 |
| 401xx | `40101` | 未登录或 Token 无效 |
| | `40102` | Access Token 已过期 |
| | `40103` | Refresh Token 无效或已过期 |
| 403xx | `40301` | 无权限(非管理员等) |
| 404xx | `40401` | 游戏不存在 |
| | `40402` | 游戏已下架 |
| | `40403` | 扑克牌不存在 |
| 423xx | `42301` | 账户已锁定(登录失败过多) |
| 429xx | `42901` | 验证码发送太频繁 |
| | `42902` | 今日游戏次数已用完 |
| | `42903` | 成绩上报间隔太短 |
| 500xx | `50001` | 服务器内部错误 |

统一响应体:`{"code": int, "message": str, "data": any}`。

## Q2. JWT 过期时间

- Access Token = **24 小时**(原《安全规范》2.2 节的 7 天作废)
- Refresh Token = **30 天**
- 算法 HS256;前端剩余时间不足时用 Refresh 静默续期

## Q3. 密码登录

- MVP **不实现**密码登录与找回密码
- 数据库 `users.password_hash` 字段**保留预留**,第三阶段视需要再启用
- 登录方式:仅手机号 + 短信验证码

## Q4. CSRF

- 纯前后端分离 + JWT(Bearer Header),**不使用 Cookie 会话**
- 因此**不引入 CSRF Token**,《安全规范》6.3 节作废
- 保留 6.4 节其余安全响应头(已在 `app.py` 实现)

## Q5. 验证码发送与登录锁定解耦

- 发码限流:同手机号 60 秒 1 次(独立逻辑)
- 登录锁定:同手机号 15 分钟内验证码错误 ≥5 次 → 锁定 15 分钟(独立逻辑)
- 锁定期间允许发码,但登录校验直接拒绝,返回 `42301`

## Q6. 积分结算归属

- 游戏插件仅上报原始成绩(`score`/`duration`/胜负等)
- **积分换算由公共服务层统一执行**,按各游戏 `config.json` 的
  `points_per_win`/`points_per_loss` 计算
- 游戏插件无权直接读写积分或数据库(与《安全规范》第八节一致)

## Q7. 数据库文件路径

- 生产:`/data/poker/poker.db`(代码目录外),每日 3:00 备份留 7 天
- 本地开发:项目内 `backend/poker_dev.db`(`DevConfig` 默认,git 忽略)
