> 面向**游戏前端开发者**。以蜘蛛纸牌的完整开发流程为参考，沉淀出可复用的模式。

>

## 1. 核心概念：游戏页面状态机

每个游戏页面必须实现以下 **5 种状态**：

```
ready  →  loading  →  playing  →  won / lost
(选难度/       (调 API,       (交互中)      (结算)
 点开始)       等数据)
```

| 状态 | UI 内容 | API 调用 | 规则弹窗 |
|------|---------|---------|---------|
| `ready` | 游戏图标 + 名称 + 难度选择 + [开始游戏] | ❌ 不调 | ❌ 不显示 |
| `loading` | "发牌中…" | ✅ `/play` 请求中 | ❌ |
| `playing` | 牌面 + 信息栏 + 操作按钮 | — | ✅ 首次自动弹出 |
| `won` | 🎉 恭喜通关 + 统计 + [再来一局] | `/score` 已上报 | — |
| `lost` | 😔 游戏结束 + 统计 + [再来一局] | `/score` 已上报 | — |

### 1.1 页面加载预检（必须）

**每个正式游戏页面必须经统一路由守卫调用 `/check` 端点预检用户可玩性。** 当前项目采用 `GameGuard` 包裹游戏页面，而不是在每个页面内重复维护 `blocked` 状态。

| 检查项 | 端点 | 失败错误码 | 含义 |
|--------|------|-----------|------|
| 游戏存在且已上架 | `POST /api/games/<id>/check` | `40401` / `40402` | 游戏不存在或已下架 |
| 今日次数未用完 | 同上 | `42902` | 今日游戏次数已用完 |

**为什么必须在进入页面前检查？**
- `/check` 是只读端点，不消耗每日游戏次数
- 用户打开页面看一眼就走，不应被扣次数
- 把"无法游戏"的警告放在页面入口层，避免每个游戏页面重复实现拦截逻辑

**标准实现模式：**

```jsx
// frontend/src/config/gameRegistry.js
{
  gameId: 'game_id',
  route: '/game/game_id',
  component: GamePage,
}

// App.jsx
<Route
  path={route}
  element={<GameGuard gameId={gameId}><GamePage /></GameGuard>}
/>
```

> **注意**：`GameGuard` 负责未登录、已下架、次数耗尽三类阻断页。游戏页面自己的 `ready` 状态只处理玩法准备，例如难度选择、开始按钮和规则说明，不再重复做 `/check` 预检。


### 状态转换代码骨架

```jsx
const [state, setState] = useState('ready')

if (state === 'ready') return <ReadyScreen />
if (state === 'loading') return <LoadingScreen />
if (state === 'error') return <ErrorScreen />
if (state === 'won' || state === 'lost') return <SettlementScreen />
return <PlayingScreen />   // state === 'playing'
```

> **重要**：`ready` 状态下不要调用任何 API（包括 `/play`），否则会消耗每日游戏次数。
> 这个 bug 在开发早期多次出现——用户只是打开页面看一眼就走了，却被扣了次数。

## 3. 状态管理：Ref 模式

### 3.1 为什么必须用 ref

React 中 `useCallback` 会**捕获创建时的闭包值**。如果回调函数依赖 `columns` state，快速连击时会拿到旧的 `columns`：

```js
// ❌ 错误：columns 来自闭包，可能过期
const moveCard = useCallback((src, dst) => {
  const sourceCol = columns[src]    // ← 可能是 0.2 秒前的值
  ...
}, [columns])  // 每次 columns 变化，moveCard 才刷新
```

**现象**：200 步自动操作后，board 状态完全没变。蜘蛛纸牌曾因此浪费 2 小时调试。

### 3.2 标准写法

```js
// ✅ 正确：用 ref 桥接，永远拿最新值
const columnsRef = useRef(columns)
const stockRef = useRef(stock)

useEffect(() => { columnsRef.current = columns }, [columns])
useEffect(() => { stockRef.current = stock }, [stock])

const moveCard = useCallback((src, dst) => {
  const cols = columnsRef.current   // ← 永远是最新值
  ...
}, [])  // 依赖数组可以为空！
```

**规则**：所有读取 `columns` / `stock` 的游戏操作函数（moveCard、dealStock、checkDeadEnd 等），**一律从 ref 读**，不从 state 读。

### 3.3 ref 依赖清单

| 数据类型 | state | ref | ref sync useEffect |
|---------|-------|-----|---------------------|
| `columns` — 列数据 | `useState([])` | `columnsRef` | `useEffect` |
| `stock` — 手牌堆 | `useState([])` | `stockRef` | `useEffect` |
| `selected` — 选中态 | `useState(null)` | 不需要 ref | — |
| `flippingIds` — 动画追踪 | `useState(Set)` | 不需要 ref | — |

## 4. 点击交互：选中 / 移动分离模式

### 4.1 为什么必须拆分

蜘蛛纸牌最初的 `handleCardClick` 把"选中牌"和"尝试移动"写在同一个函数里，导致：

1. 点牌 A 选中它 → 没弹 toast ✅
2. 点牌 B 想移动 → 失败弹 "此牌不能放这里" ✅（合法）
3. 但用户刚看清楚牌面，随手一点就弹红色 toast，体验极差 ❌

### 4.2 标准拆分

```js
// ✅ 纯选择 — 永不弹 toast
const selectCard = useCallback((colIdx, cardIdx) => {
  const col = columnsRef.current[colIdx]
  const group = getDraggableGroup(col, cardIdx)
  if (group) setSelected({ col: colIdx, idx: cardIdx })
}, [])

// ✅ 尝试移动 — 失败才弹 toast
const tryMove = useCallback((targetCol, targetCardIdx) => {
  if (!selected) return

  // 同一列 → 切换选中或取消
  if (selected.col === targetCol) {
    if (selected.idx === targetCardIdx) setSelected(null)
    else selectCard(targetCol, targetCardIdx)
    return
  }

  // 不同列 → 校验并移动
  const group = columnsRef.current[selected.col].slice(selected.idx)
  const target = columnsRef.current[targetCol]
  if (isValidSequence(group) && canDrop(group[0], target.at(-1))) {
    moveCards(selected.col, selected.idx, targetCol)
  } else {
    showToast('此牌不能放这里')
    setSelected(null)
  }
}, [selected, showToast])

// ✅ 总入口 — 只做分派
const handleCardClick = useCallback((colIdx, cardIdx) => {
  if (!columnsRef.current[colIdx][cardIdx].faceUp) return
  if (selected) tryMove(colIdx, cardIdx)
  else selectCard(colIdx, cardIdx)
}, [selected, selectCard, tryMove])
```

### 4.3 点击事件的 DOM 层级

```jsx
{/* 列容器 — 点击空列区域 */}
<div onClick={() => {
  if (col.length === 0) handleEmptyColumnClick(idx)
  else if (selected) handleColumnAreaClick(idx, col)
}}>
  {/* 单张牌 — stopPropagation 阻止冒泡到列 */}
  <div onClick={(e) => {
    e.stopPropagation()
    if (card.faceUp) handleCardClick(colIdx, cardIdx)
  }}>
    <GameCard.Face card={card} />
  </div>
</div>
```

**规则**：牌的 onClick 必须 `e.stopPropagation()`；列容器的 onClick 处理空列点击和列背景点击。

### 5.2 JS 追踪逻辑

```js
const [flippingIds, setFlippingIds] = useState(new Set())
const flipTimerRef = useRef(null)

const triggerFlip = useCallback((cardId) => {
  setFlippingIds(prev => { const next = new Set(prev); next.add(cardId); return next })
  clearTimeout(flipTimerRef.current)
  flipTimerRef.current = setTimeout(() => setFlippingIds(new Set()), 500)
}, [])

// 在 moveCards 中：牌被移走后露出背面牌时
if (uncovered) {
  card.faceUp = true
  triggerFlip(card.id)
}

// 在渲染中：
<div className={`... ${flippingIds.has(card.id) ? 'card-reveal' : ''}`}>
```

**规则**：`triggerFlip` 在 `moveCards` 和自动完成序列中都要调用。

## 6. 功能模块配方

### 6.1 撤销系统

```js
const [undoStack, setUndoStack] = useState([])
const MAX_UNDO = 50

// 每次操作前保存快照
const pushUndo = useCallback(() => {
  setUndoStack(prev => {
    const snap = { columns: columnsRef.current, stock: stockRef.current, moves, completed }
    const next = [...prev, snap]
    return next.length > MAX_UNDO ? next.slice(-MAX_UNDO) : next
  })
}, [moves, completed])

// moveCards 和 dealStock 开头都调用 pushUndo()
// 然后再 setColumns/setStock

// 撤销
const undo = useCallback(() => {
  if (undoStack.length === 0) return
  const last = undoStack[undoStack.length - 1]
  setColumns(last.columns)
  setStock(last.stock)
  setMoves(last.moves)
  setCompleted(last.completed)
  setUndoStack(prev => prev.slice(0, -1))
  setSelected(null)
}, [undoStack])
```

### 6.2 难度选择

```js
const [difficulty, setDifficulty] = useState(null)  // 'easy' | 'hard'

// Deck 构造 — 按难度分支
const deck = difficulty === 'easy'
  ? createDeck('S', 8)                   // 全黑桃，1 花色
  : [...createDeck('S', 2), ...createDeck('H', 2),
     ...createDeck('C', 2), ...createDeck('D', 2)]  // 4 花色

// 通关条件 — 按难度分支
const winGoal = difficulty === 'easy' ? 3 : 8

// ready 界面：两个按钮选难度，start 按钮 disabled 直到选中
// [🟢 简单] [🔴 困难]  →  [开始游戏]
```

**规则**：难度选择在 `ready` 状态中完成，选完再调 `/play`。不要在 `playing` 状态内切换难度——那意味着重新发牌。

### 6.3 死局检测

```js
function hasAnyValidMove(columns, stock) {
  if (stock.length >= 10) return true   // 还能发牌就不算死
  for (let src = 0; src < columns.length; src++) {
    const group = getDraggableGroup(columns[src], columns[src].length - 1)
    if (!group) continue
    for (let dst = 0; dst < columns.length; dst++) {
      if (dst === src) continue
      if (columns[dst].length === 0) return true
      if (canDrop(group[0], columns[dst][columns[dst].length - 1])) return true
    }
  }
  return false
}

// useEffect 检测
useEffect(() => {
  if (state !== 'playing') return
  if (!hasAnyValidMove(columnsRef.current, stockRef.current)) {
    const msg = stockRef.current.length < 10
      ? '手牌已用完且无可移动操作，请认输结束对局'
      : '当前无可移动操作，点击「发牌」补充新牌'
    showToast(msg)
  }
}, [columns, stock, state])
```

### 6.4 Toast 通知

```js
const [toast, setToast] = useState(null)
const toastTimer = useRef(null)

const showToast = useCallback((msg) => {
  setToast(msg)
  clearTimeout(toastTimer.current)
  toastTimer.current = setTimeout(() => setToast(null), 1500)
}, [])

// 渲染：
{toast && (
  <div className="fixed top-1/3 left-1/2 -translate-x-1/2 z-50
    bg-schoolred text-white px-5 py-2.5 rounded-xl shadow-lg
    text-sm font-bold text-center pointer-events-none">
    {toast}
  </div>
)}
```

```jsx

         onClick={e => e.stopPropagation()}>
      <h3>规则</h3>
      <div className="text-sm">
        {/* 规则 + 按难度自适应 */}
        {/* 游玩指南 — 独立区域 */}
        <div className="border-t pt-3 mt-3">
          <p className="font-bold">💡 游玩指南</p>
          <ol>
            <li>尽量往同一列堆牌，建立长序列</li>
```

- 规则内容按难度自适应（如简单模式提示"全黑桃，不用考虑花色"）

## 8. 信息栏设计

**禁止把操作按钮放在牌面下方单独一行**——玩家需要滚动才能看到。按钮应放在信息栏内，永远可见。

```jsx
{/* ✅ 正确：统计 + 按钮同在一行 */}
<div className="flex justify-center px-3 py-2">
  <div className="bg-white rounded-xl shadow-sm px-4 py-1.5
    flex items-center gap-3 text-xs">
    <span>🕷 {completed}/{winGoal}</span>
    <span className="text-slate-300">|</span>
    <span>⏱ {formatTime(elapsed)}</span>
    <span className="text-slate-300">|</span>
    <span>步 {moves}</span>
    <span className="text-slate-300">|</span>
    <button onClick={dealStock} disabled={stock.length < 10}
      className={`px-3 py-1 rounded-lg font-bold
        ${stock.length >= 10
          ? 'bg-amber-500 text-white animate-pulse'
          : 'bg-slate-100 text-slate-300 cursor-not-allowed'}`}>
      {stock.length >= 10 ? `发牌 ${stock.length/10}轮` : '手牌空'}
    </button>
    <button onClick={undo} disabled={undoStack.length === 0}>
      ↩ 撤销
    </button>
  </div>
</div>
```

**规范**：
- 进度、倒计时、步数、操作按钮**都在同一行白底卡片中**
- 发牌按钮：有牌可发时 amber 脉冲，无牌时灰显
- 撤销按钮：有历史时正常，无历史时灰显
- 不要出现"只显示几项，另外几项在底部"的情况

## 13. 常见陷阱与修复

| 现象 | 根因 | 修复参考 |
|------|------|---------|
| 未登录或次数耗尽时直接进到游戏页 | 缺少 `GameGuard` 路由守卫 | 第 1.1 节 — 状态机 |
| 快速连点，牌没移过去 | `columns` 来自过期闭包 | 第 3 节 — Ref 模式 |
| 第一次点牌就弹 toast "不能放" | 选中和移动逻辑混写 | 第 4 节 — 分离模式 |
| 背面牌直接"闪现"成正面 | 缺少翻面动画 | 第 5 节 — 翻牌动画 |
| 发牌按钮找不到 | 按钮放在可滚动区域底部 | 第 8 节 — 信息栏 |
| 撤销后步数没回退 | 撤销时只恢复了 columns，没恢复 moves | 第 6.1 节 — 撤销 |
| 难度选了没生效 | `handleStart` 没把 `difficulty` 加入依赖 | 第 6.2 节 — 难度 |
| 玩家不知道已经无解 | 缺少死局检测 | 第 6.3 节 — 死局检测 |
| 简单模式也要集齐 8 组 | winGoal 硬编码为 8 | 第 6.2 节 — 难度 |
