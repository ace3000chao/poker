## 二、JWT Token 规范

### 2.1 Token 结构

采用双Token机制：
- **Access Token**：短期Token，用于API认证
- **Refresh Token**：长期Token，用于续期Access Token

### 2.2 Token 配置

| 参数 | 值 |
|------|---|
| Access Token 过期时间 | **7天** |
| Refresh Token 过期时间 | **30天** |
| Token 加密算法 | HS256 |
| Token 存储位置 | 客户端 localStorage 或 memory |

### 2.3 Token Payload 结构

```json
{
  "sub": "user_id",
  "phone": "13800138000",
  "role": "user",
  "type": "access",
  "iat": 1715840000,
  "exp": 1716444800
}
```

Refresh Token Payload：
```json
{
  "sub": "user_id",
  "type": "refresh",
  "iat": 1715840000,
  "exp": 1718441600
}
```

### 2.4 Token 刷新流程

```
1. 用户登录，获取 Access Token + Refresh Token
2. 前端每次请求带 Access Token
3. Access Token 即将过期时（前端自行判断，剩余 < 1天），用 Refresh Token 换取新的 Access Token
4. Refresh Token 过期后，需要重新登录
```

### 2.5 Token 续期接口

```
POST /api/auth/refresh
```

**请求头**：
```
Authorization: Bearer {refresh_token}
```

**响应示例**：
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "access_token": "eyJhbGciOiJIUzI1NiIs...",
    "expires_in": 604800
  }
}
```

**错误码**：
| code | 说明 |
|------|------|
| 0 | 成功 |
| 40102 | Refresh Token 已过期 |
| 40103 | Refresh Token 无效 |

## 三、短信验证码规范

### 3.1 发送验证码

```
POST /api/auth/send-code
```

**请求体**：
```json
{
  "phone": "13800138000",
  "purpose": "login"
}
```

`purpose` 可选值：
- `login`：登录
- `register`：注册
- `reset_password`：找回密码

**频率限制**：
- 同一手机号 **60秒内只能发送1次**
- 验证码 **5分钟** 内有效
- 验证成功后验证码立即失效

**响应示例**：
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "expires_in": 300,
    "next_can_send_in": 60
  }
}
```

**错误码**：
| code | 说明 |
|------|------|
| 0 | 成功 |
| 40001 | 参数格式错误 |
| 42901 | 发送太频繁，请稍后再试 |

### 3.2 验证码登录/注册

```
POST /api/auth/login
```

**请求体**：
```json
{
  "phone": "13800138000",
  "code": "123456",
  "purpose": "login"
}
```

**响应示例**：
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "access_token": "eyJhbGciOiJIUzI1NiIs...",
    "refresh_token": "eyJhbGciOiJIUzI1NiIs...",
    "expires_in": 604800,
    "refresh_expires_in": 2592000,
    "user": {
      "id": 1,
      "phone": "13800138000",
      "role": "user",
      "points": 0
    }
  }
}
```

**错误码**：
| code | 说明 |
|------|------|
| 0 | 成功 |
| 40001 | 参数格式错误 |
| 40002 | 验证码错误 |
| 40003 | 验证码已过期 |
| 40004 | 验证码已使用 |
| 40005 | 目的不匹配（如用login验证码去register） |
| 42901 | 登录失败次数过多，账户已锁定 |

## 四、登录安全规范

### 4.1 登录失败锁定机制

| 参数 | 值 |
|------|---|
| 失败次数上限 | **5次** |
| 锁定时间 | **15分钟** |
| 锁定触发 | 连续5次验证码/密码错误 |
| 解锁方式 | 15分钟后自动解锁 |

### 4.2 登录流程

```
1. 用户输入手机号，点击获取验证码
2. 系统检查：该手机号最近15分钟内失败次数是否 >= 5
   - 是：拒绝发送，返回错误码 42901
   - 否：继续下一步
3. 发送验证码，记录发送时间
4. 用户输入验证码，提交登录
5. 系统检查：验证码是否正确、过期、已使用
   - 失败：记录失败次数，返回错误码 40002/40003/40004
   - 成功：重置失败次数，返回 Token
```

### 4.3 登录接口

```
POST /api/auth/login
```

### 4.4 错误码

| code | 说明 |
|------|------|
| 0 | 成功 |
| 40001 | 参数格式错误 |
| 40002 | 验证码错误 |
| 40003 | 验证码已过期 |
| 40004 | 验证码已使用 |
| 40005 | 目的不匹配 |
| 42901 | 账户已锁定，请15分钟后再试 |

## 五、密码安全规范（可选）

### 5.1 注册时设置密码（可选）

用户注册时可不设置密码，之后可通过"忘记密码"设置。

### 5.2 密码要求

| 要求 | 规则 |
|------|------|
| 最小长度 | 8位 |
| 必须包含 | 大小写字母 + 数字 |
| 禁止 | 纯数字、纯字母、连续字符（12345678） |

### 5.3 密码存储

使用 **bcrypt** 哈希算法存储：

```python
import bcrypt

password_hash = bcrypt.hashpw(password.encode(), bcrypt.gensalt())
bcrypt.checkpw(password.encode(), password_hash)
```

### 5.4 忘记密码流程

```
POST /api/auth/reset-password/send-code  # 发送验证码到手机
POST /api/auth/reset-password  # 提交新密码
```

## 六、Web 安全防护

### 6.1 SQL 注入防护

- 使用 **SQLAlchemy ORM**，所有查询通过参数化查询
- 禁止拼接 SQL 字符串
- 用户输入必须经过验证和清洗

### 6.2 XSS 防护

- 用户输入存储前进行 HTML 转义
- 富文本内容使用 DOMPurify 过滤
- HTTP响应头添加 `X-XSS-Protection`

### 6.3 CSRF 防护

- 使用 Flask-WTF 生成 CSRF Token
- 所有 POST/PUT/DELETE 请求必须带 CSRF Token
- Token 放在请求头或表单中

### 6.4 其他安全头

```python
# Flask 配置
@app.after_request
def add_security_headers(response):
    response.headers['X-Content-Type-Options'] = 'nosniff'
    response.headers['X-Frame-Options'] = 'DENY'
    response.headers['X-XSS-Protection'] = '1; mode=block'
    response.headers['Strict-Transport-Security'] = 'max-age=31536000; includeSubDomains'
    return response
```

## 八、第三方游戏安全要求

第三方游戏接入时必须遵守：

1. **禁止** 直接访问数据库，必须通过 API
2. **禁止** 在游戏代码中存储用户敏感信息
3. **必须** 使用游戏插件接口规范定义的标准接口
4. **必须** 在 `config.json` 中声明需要的数据权限
