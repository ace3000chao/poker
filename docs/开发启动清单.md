## 二、待澄清问题 + 建议答案(请逐项确认)

### Q1. 错误码体系统一【建议:重建全局唯一错误码表】
现状:`42901`/`42902` 在"账户锁定/发码频繁/今日次数用完/上报间隔短"等多场景被复用,冲突。
**建议**:按模块分段,全局唯一:
- `0` 成功
- `400xx` 参数/业务错误
- `401xx` 认证错误(未登录/Token 过期)
- `403xx` 权限错误
- `404xx` 资源不存在(游戏/牌)
- `429xx` 限流(发码频繁=42901、今日次数用完=42902、上报间隔短=42903)
- `423xx` 锁定(账户锁定=42301)
- `500xx` 服务器错误

→ 确认后我会同步修订三份技术文档的错误码表。

### Q2. JWT 过期时间【✅ 已定:Access 24 小时,Refresh 30 天】
安全规范初稿定 Access 7 天偏长。**已确认**:Access Token = **24 小时**(当天有效,次日续期),Refresh Token = 30 天,前端用 Refresh 静默续期。→ 修订安全规范第 2.2 节。

### Q3. 密码登录是否纳入 MVP【✅ 已定:MVP 不做,仅手机号+验证码】
**已确认**:数据库 `password_hash` 字段**保留(预留)**,MVP **不实现**密码登录与找回密码。第三阶段视需要再加。

### Q4. CSRF 与 JWT 并用【建议:纯 API + JWT,不引入 CSRF Token】
现状:安全规范同时要求 JWT(localStorage)+ Flask-WTF CSRF Token,二者并用是矛盾设计。
**建议**:前后端分离纯 API 架构,认证只用 JWT(Bearer Header),**不使用 Cookie 会话,故不需要 CSRF Token**。安全规范第 6.3 节相应修订。

### Q5. 验证码与登录锁定的耦合【建议:发码与登录分别独立限流】
**建议**:
- 发码限流:同手机号 60 秒 1 次(独立)
- 登录锁定:同手机号 15 分钟内验证码错误 ≥5 次 → 锁定 15 分钟(独立,不影响已通过的发码)
- 锁定期间:可发码但登录校验直接拒绝,返回 `42301`

### Q6. 积分规则归属【建议:平台统一结算,游戏只上报原始成绩】
现状:PRD 说"积分规则由各游戏定义",插件规范又把 `points_per_win/loss` 放在 config.json。
**建议**:游戏只通过 `score` 字段上报成绩 + 胜负标志,**积分换算由公共服务层统一按 config.json 的 `points_per_win/loss` 计算**,游戏插件无权直接写积分(契合安全规范"禁止游戏直接访问数据库")。

### Q7. 数据库文件路径【建议:沿用 `/data/poker/poker.db`,代码目录外】
确认采用,部署时确保该目录权限与备份 cron(每日 3:00,留 7 天)就位。
